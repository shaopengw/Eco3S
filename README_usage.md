# AI 辅助模拟实验系统使用说明

本文档详细介绍了如何使用 AI 辅助模拟实验系统。该系统旨在通过自然语言需求，自动化地完成从实验设计、代码生成、模拟运行到结果评估与优化的全过程。

## 系统核心工作流

系统通过一系列智能体（Agent）协同工作，将一个高层次的模拟需求转化为可执行的代码和可分析的结果。核心流程分为以下几个阶段：

![系统工作流](https://i.imgur.com/your-workflow-diagram.png)  <!-- 占位符，可替换为实际流程图 -->

### 阶段 1: 需求分析与项目初始化

- **输入**: 用户通过命令行输入自然语言描述的模拟需求。
- **处理**: `ProjectMasterAgent` 解析需求，提取关键信息，如模拟名称、核心目标等。
- **输出**:
  - 创建独立的配置文件夹 (`config/<simulation_name>/`)。
  - 创建独立的实验数据文件夹 (`history/<simulation_name>/`)。

### 阶段 2: 系统设计

- **处理**: `SimArchitectAgent` 根据解析后的需求，进行系统设计。
- **输出**:
  - **设计文档 (`description.md`)**: 详细描述实验目标、模拟实体、环境、关键行为和预期结果。
  - **模块配置文件 (`modules_config.yaml`)**: 定义了本次模拟需要启用的功能模块（如气候、经济、叛乱等）及其高级配置。

### 阶段 3: 代码与配置生成

- **处理**: `CodeArchitectAgent` 根据设计文档和模块配置，生成所有必需的代码和配置文件。
- **亮点**:
  - **细粒度生成**: 每个文件（模拟器、主函数、配置文件、提示词）都独立生成，降低上下文过载风险。
  - **循环重试与诊断**:
    - 对代码生成的每个子步骤（如识别未实现函数、补完函数）都设置了循环重试机制。
    - 若多次重试失败，将触发大模型进行**智能诊断**，分析失败的根本原因并提供解决方案。
- **输出**:
  - **模拟器代码**: `src/simulation/simulator_<simulation_name>.py`
  - **主入口文件**: `entrypoints/main_<simulation_name>.py`
  - **详细配置文件**: `simulation_config.yaml`, `jobs_config.yaml` 等。
  - **Agent提示词**: `residents_prompts.yaml`, `government_prompts.yaml` 等。

### 阶段 4: 模拟运行与自动纠错

- **处理**: `ProjectMasterAgent` 尝试执行生成的主入口文件。
- **亮点**:
  - **自动纠错**: 如果程序在运行时出错（例如，语法错误、依赖缺失），系统会捕获错误信息，并自动调用 `CodeArchitectAgent` 进行修复。此过程会重试多次，直到成功运行或达到最大尝试次数。
- **输出**:
  - **模拟结果数据**: 通常是 `.csv` 或 `.json` 文件，保存在 `history/<simulation_name>/[具体模拟名称/模拟时间]` 目录下。
  - **运行日志**: 详细的运行过程日志。

### 阶段 5: 结果评估与自动优化

- **处理**: `ResearchAnalystAgent` 分析模拟运行的结果。
- **亮点**:
  - **自动评估**: 将模拟结果与设计文档中的预期目标进行比对，判断模拟是否成功。
  - **诊断与优化**:
    1. 如果结果不符合预期 (`NEED_ADJUSTMENT`)，Agent会诊断问题根源（是参数配置不当还是提示词设计问题）。
    2. 接着，它会自动生成修改方案，并**精确地修改**相关配置文件的参数值，而**不破坏文件结构**。
  - **循环优化**: 修改配置后，系统会自动返回**阶段 4**，重新运行模拟，形成一个“**运行-评估-优化**”的闭环，直到结果符合预期或达到最大迭代次数。

## 如何运行系统

### 1. 启动系统

在项目根目录下运行以下命令：

```bash
python run_ai_system.py
```

### 2. 选择运行模式

系统提供两种运行模式，以适应不同需求：

```
请选择运行模式：
  1. 自动模式 - 一次性完成所有步骤，自动循环优化。
  2. 交互模式 - 每个主要阶段后等待用户确认，提供手动干预机会。

请输入选项 (1/2，默认为1):
```

#### 自动模式 (推荐)

- **特点**: 一键到底，从需求输入到最终优化报告生成，全程无需人工干预。
- **适用场景**:
  - 需求明确，希望快速得到最终结果。
  - 运行完整的端到端测试。

#### 交互模式

- **特点**: 在“设计”、“编码”、“运行与优化”等关键阶段后会暂停，等待用户确认。用户可以审查中间产物（如设计文档、生成的代码），并决定是继续、退出还是提供反馈以重新生成。
- **适用场景**:
  - 需求探索，需要逐步验证和调整。
  - 教学演示，清晰展示系统每一步的工作。
  - 对特定阶段的结果进行微调。

### 3. 输入模拟需求

根据提示，用自然语言输入您的模拟需求。输入回车结束。

**示例:**
```
请输入您的模拟实验需求:
我希望研究在一个有极端气候的世界中，政府如何通过调整税收和运河维护投资来平衡财政并抑制叛乱。模拟中应包含政府、居民和叛军三种角色，极端气候会随机发生并对运河造成破坏。[输入回车结束]
```

## 输出结果

无论在哪种模式下，成功运行后，您都可以在以下目录中找到所有生成的文件：

- **配置文件**: `config/<simulation_name>/`
  - `description.md`: AI生成的设计文档。
  - `modules_config.yaml`: 启用的功能模块。
  - `simulation_config.yaml`: 模拟核心配置。
  - `jobs_config.yaml`: 任务配置。
  - `resident_actions.yaml`: 居民行为配置。
  - `towns_data.json`: 城镇数据。
  - `government_prompts.yaml`: 政府Agent提示词。
  - `rebels_prompts.yaml`: 叛军Agent提示词。
  - `residents_prompts.yaml`: 居民Agent提示词。
  - `*.yaml`, `*.json`: 其他详细配置和提示词。

- **源代码**:
  - `src/simulation/simulator_<simulation_name>.py`: 核心模拟器逻辑。
  - `entrypoints/main_<simulation_name>.py`: 运行模拟的入口脚本。

- **实验数据与结果**: `history/<simulation_name>/`

## 常见问题 (FAQ)

**Q: 交互模式可以返回到上一个阶段吗？**  
A: 当前版本不支持从“编码”阶段返回“设计”阶段。但您可以在当前阶段内，通过提供反馈意见来多次重新生成内容。

**Q: 自动优化循环会运行多少次？**  
A: 默认最多运行3次，您可以在 `project_master.py` 的 `run_full_workflow` 方法中修改 `max_iterations` 参数。

**Q: 如果代码生成或运行修复连续失败怎么办？**  
A: 系统会在多次尝试失败后，提供详细的诊断日志，指出根本原因。您可以根据此日志手动修复问题，或优化您的初始需求描述。

**Q: 我可以直接修改生成的代码或配置吗？**  
A: 当然可以。AI生成的文件是标准的Python代码和YAML/JSON文件，您可以随时在生成的代码基础上进行手动修改和扩展。
